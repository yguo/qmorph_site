## 2. 技术选型

### 2.1 核心组件对比表

#### STT (Speech-to-Text)

| 方案 | 延迟 | 准确率 | 价格(/分钟) | 多语言 | 医疗术语 |
|------|------|--------|------------|--------|---------|
| **Deepgram Nova-3** | ~250ms | 97%+ | $0.0043 | 是 | 是 |
| OpenAI Whisper API | ~800ms | 96% | $0.006 | 是 | 一般 |
| Google Speech-to-Text | ~400ms | 95% | $0.016 | 是 | 一般 |
| AssemblyAI | ~500ms | 96% | $0.0065 | 部分 | 一般 |

**推荐 Deepgram Nova-3 理由：** 流式实时转录延迟业界最低，内置 VAD（语音活动检测），支持医疗/法律专业词汇模型，Streaming WebSocket API 与 Telnyx 原生集成，成本比 Google 低 73%。

#### TTS (Text-to-Speech)

| 方案 | 延迟(首包) | 自然度 | 价格(/字符) | 克隆声音 | 情绪控制 |
|------|-----------|--------|------------|---------|---------|
| **Cartesia Sonic** | ~90ms | 极高 | $0.000065 | 是 | 是 |
| ElevenLabs | ~300ms | 极高 | $0.00024 | 是 | 是 |
| OpenAI TTS | ~400ms | 高 | $0.000015 | 否 | 否 |
| Azure Neural TTS | ~200ms | 高 | $0.000016 | 部分 | 部分 |

**推荐 Cartesia Sonic 理由：** 首包音频延迟低至 90ms（业界最快），支持实时流式输出，声音克隆可为每个客户定制前台音色，情绪/语速控制适合专业场景，价格适中。

#### LLM (Language Model)

| 方案 | 延迟(TTFT) | 理解能力 | 工具调用 | 价格(/1M tokens) | 上下文窗口 |
|------|-----------|---------|---------|-----------------|----------|
| **Claude 3.5 Sonnet** | ~300ms | 极强 | 优秀 | $3/$15 | 200K |
| GPT-4o | ~350ms | 极强 | 优秀 | $2.5/$10 | 128K |
| GPT-4o-mini | ~150ms | 强 | 良好 | $0.15/$0.6 | 128K |
| Gemini 1.5 Flash | ~200ms | 强 | 良好 | $0.075/$0.3 | 1M |

**推荐 Claude 3.5 Sonnet 理由：** 指令遵循能力业界最强，Tool Use（工具调用）JSON结构化输出稳定，对敏感对话（医疗/法律）处理更审慎，200K上下文可容纳完整业务知识库，Anthropic对企业级合规支持完善（BAA协议可签）。

#### 电话基础设施

| 方案 | WebSocket流 | 通话录音 | 号码类型 | 价格(/分钟) | SIP Trunk |
|------|------------|---------|---------|------------|---------|
| **Telnyx** | 是 | 是 | DID/TFN | $0.002 | 是 |
| Twilio | 是 | 是 | DID/TFN | $0.0085 | 是 |
| Vonage | 部分 | 是 | DID | $0.004 | 是 |
| SignalWire | 是 | 是 | DID | $0.003 | 是 |

**推荐 Telnyx 理由：** TeXML + WebSocket 媒体流原生支持，延迟比 Twilio 低约 40%，价格比 Twilio 低 76%，自建全球网络骨干（非转售），提供 STIR/SHAKEN 认证防止被标记为骚扰电话，支持紧急呼叫（E911）。

## 3. 核心功能模块

### 3.1 智能路由引擎 (Intelligent Call Router)

```
来电 → 意图识别 → 路由决策树
         │
         ├── 紧急情况 → 立即转人工 / 拨打911
         ├── 预约请求 → 预约引擎
         ├── 账单查询 → 查询CRM数据 → 语音回答
         ├── 转接指定人员 → SIP转接
         ├── 营业时间外 → 语音信箱 / 短信回调
         └── 无法处理 → 转人工 + 通话摘要推送
```

核心能力：
- 基于 LLM 的意图分类（30+ 意图类别，可按行业扩展）
- 动态优先级队列（VIP客户、回访患者识别）
- 并发通话管理（每租户独立并发限制）
- 转接时自动生成通话摘要，推送给接听人员
- 溢出保护：超并发自动排队 + 估计等待时间播报

### 3.2 预约引擎 (Appointment Scheduling Engine)

```
用户:"我想预约下周二下午看牙"
     │
     ▼
意图提取 → 日期/时间/服务类型/医生偏好解析
     │
     ▼
可用性查询 → 读取日历API (Google Calendar / Dentrix / Clio)
     │
     ▼
时间槽推荐 → 最多提供3个选项 → 用户确认
     │
     ▼
预约创建 → 写入目标系统 → 发送确认短信/邮件
     │
     ▼
提醒调度 → 24小时前/2小时前自动提醒
```

核心能力：
- 自然语言时间解析（"后天""下个月第一个周一"）
- 冲突检测与自动备选推荐
- 多资源调度（医生+诊室+设备）
- 候补名单管理（取消时自动通知候补患者）
- 重复预约支持（每月定期维护等）

### 3.3 管理后台 (Admin Dashboard)

模块结构：

## 5. 多租户 SaaS 架构

### 5.1 数据隔离模型

采用 **Hybrid Multi-tenancy** 策略：

```
租户隔离级别
├── 共享数据库 + Row-Level Security (RLS)
│   适用：通话日志、分析数据、系统事件
│   实现：PostgreSQL RLS + tenant_id 行级策略
│
├── 独立 Schema
│   适用：业务数据（预约、联系人、知识库）
│   实现：每租户独立 PostgreSQL schema
│   命名：tenant_{uuid}.*
│
└── 独立实例（Enterprise 套餐）
    适用：HIPAA BAA 签署客户、超大规模
    实现：独立 RDS 实例 + 独立 S3 存储桶
```

核心隔离措施：
- 所有 API 请求经 middleware 注入 `tenant_id` 上下文
- JWT Token 携带 `tenant_id` + `role` claims
- 知识库向量数据按 `namespace` 在 Pinecone 隔离
- Redis 缓存 Key 前缀：`tenant:{id}:{resource}:{key}`
- 录音文件存储路径：`s3://recordings/{tenant_id}/{year}/{month}/{call_id}.mp3`

### 5.3 HIPAA 合规要点

适用场景：牙科、医疗相关租户（患者姓名、预约信息、保险信息属于PHI）

**技术保障措施 (Technical Safeguards)：**

| 要求 | 实现方案 |
|------|---------|
| 传输加密 | TLS 1.3（所有API）/ SRTP（音频流）|
| 静态加密 | AES-256（RDS存储加密 / S3 SSE-KMS）|
| 访问控制 | RBAC + MFA + 最小权限原则 |
| 审计日志 | 所有PHI访问操作写入不可篡改日志（CloudTrail + 自建审计表）|
| 自动登出 | 管理后台30分钟无操作强制登出 |
| 录音访问 | 签名URL（有效期15分钟）+ 访问日志 |

**管理保障措施 (Administrative Safeguards)：**
- 与租户签署 Business Associate Agreement (BAA)
- 与 Anthropic / Deepgram / Cartesia / Telnyx 签署 BAA（均支持）
- 年度安全培训记录
- 数据泄露应急响应预案（72小时通知义务）

**物理保障措施 (Physical Safeguards)：**
- 部署在 AWS GovCloud 或 AWS us-east-1（SOC2 Type II 认证）
- 开发/测试环境不使用真实PHI数据

**数据保留策略：**
- 通话录音：默认保留7年（医疗法规要求），租户可配置
- 对话文本日志：默认保留3年，HIPAA租户自动加密存储
- 员工离职后立即撤销访问权限

## 6. 延迟优化方案

### 6.1 目标与预算

**总体目标：** 用户说完话到AI开始回复 < 1500ms（感知自然对话）

```
延迟预算分解：

用户停止说话
     │
     ▼  VAD检测停顿
    +80ms  (Deepgram 端点检测)
     │
     ▼  STT 最终转录
   +220ms  (Deepgram Nova-3 流式最终结果)
     │
     ▼  LLM 处理 (TTFT - Time to First Token)
   +400ms  (Claude 3.5 Sonnet，缓存命中情况)
     │
     ▼  TTS 首包生成
    +90ms  (Cartesia Sonic 流式首包)
     │
     ▼  网络传输 + 音频缓冲
   +100ms
     │
     ▼  用户听到回复
─────────
总计: ~890ms (理想情况)
目标: <1500ms (P95)
上限: <2000ms (P99，超出则触发优化告警)
```

### 6.2 各组件延迟预算表

| 组件 | 目标延迟 | P95上限 | 优化手段 |
|------|---------|---------|---------|
| VAD 端点检测 | 80ms | 150ms | Deepgram endpointing 参数调优 |
| STT 最终转录 | 220ms | 350ms | Nova-3 模型 + interim_results 流式 |
| 网络(用户→Telnyx) | 30ms | 80ms | 就近接入节点选择 |
| 网络(Telnyx→服务) | 20ms | 50ms | 服务部署在 us-east-1，靠近Telnyx PoP |
| 意图识别+路由 | 50ms | 100ms | 规则引擎优先，LLM兜底 |
| LLM TTFT | 400ms | 700ms | Prompt缓存 + 系统提示压缩 |
| 工具调用(CRM查询) | 150ms | 300ms | Redis缓存 + 预加载常用数据 |
| TTS 首包 | 90ms | 150ms | Cartesia 流式 + 连接预热 |
| 音频编码+传输 | 50ms | 100ms | Opus编码，就近CDN |
| **端到端总计** | **890ms** | **1500ms** | — |

### 6.3 关键优化策略

**策略 1：LLM Prompt 缓存（最高优先级）**
- 系统提示（知识库+业务规则）固定化，利用 Claude 的 Prompt Caching（cache_control）
- 缓存命中可将 LLM TTFT 从 800ms 降至 400ms
- 每次对话开始时预热缓存

**策略 2：流式管道（Streaming Pipeline）**
- STT → LLM → TTS 全链路流式，无需等待上一步完成
- STT 产出第一个完整句子后立即发给 LLM
- LLM 产出第一个 token 后立即发给 TTS
- 实测可减少约 300-400ms 感知延迟

**策略 3：打断检测（Barge-in）**
- 用户说话时立即停止 TTS 播放
- VAD 实时检测，延迟 < 100ms 响应打断
- 避免用户等待 AI 说完才能插话

**策略 4：预加载与缓存**
```
来电响铃阶段（~2秒）已完成：
├── 加载租户配置和知识库
├── 建立 LLM 连接并发送系统提示（触发缓存）
├── 建立 TTS WebSocket 连接
└── 查询来电号码历史（预加载客户信息）
```

**策略 5：降级策略**
- LLM 超时 800ms：降级到规则引擎快速响应
- TTS 超时 200ms：切换备用 TTS（Azure Neural）
- CRM 查询超时 500ms：跳过，仅使用通话内信息

**策略 6：地理分布部署**
- 主部署：AWS us-east-1（覆盖美东主要客户）
- 次部署：AWS us-west-2（西海岸客户）
- 后续：AWS eu-west-1（欧洲扩展）
- 根据来电区号智能路由到最近服务节点

### 6.4 延迟监控指标

持续追踪以下 P50/P95/P99 指标：
- `latency.stt.final` — STT最终转录延迟
- `latency.llm.ttft` — LLM首token延迟
- `latency.tts.first_audio` — TTS首包延迟
- `latency.e2e.response` — 端到端响应延迟（最关键）
- `latency.tool_call.crm` — CRM工具调用延迟
- `call.interruption_rate` — 打断发生率（>15%说明AI回复过慢）

告警阈值：P95端到端延迟 > 1500ms 触发 PagerDuty。
